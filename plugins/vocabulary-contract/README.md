# vocabulary-contract

A Claude Code plugin with one skill:
**[vocabulary-contract](skills/vocabulary-contract/SKILL.md)** — turns a
`termbase.csv` controlled vocabulary into an enforceable naming contract for
spec-driven development (proposal → spec → tasks → implementation).

It needs only the CSV — it does not depend on whatever produced it. A
`termbase.csv` from this repo's `terminology-audit` plugin works directly,
and so does a hand-maintained one, as long as it has at minimum a `term`
column (see the skill's tolerant column-mapping for accepted header names).

Three modes:

1. **Render** — generate an AGENTS.md / CLAUDE.md vocabulary fragment
   (imperative prose: canonical term, banned variants, naming-transform per
   surface — code identifier, file name, CLI flag, env var) plus a compact
   spec-preamble block.
2. **Enforce** — check a draft spec, proposal, or tasks file against approved
   terms; advisory findings (file, line, quoted sentence, canonical
   replacement, suggested rewrite), never a hard fail. A clean pass is
   reported explicitly.
3. **Round-trip** — after an implementation, surface recurring domain terms
   not yet in the termbase as candidates (with counts and locations) for the
   next revision. The termbase itself is never modified — candidates are
   proposed, a human approves.

Deterministic matching (variant detection, surface-form checks, candidate
frequency shortlisting) lives in `skills/vocabulary-contract/scripts/check_terms.py`;
interpreting the JSON and writing the human-readable report is left to the
model.

```
/plugin marketplace add JoyceMarieStack/ia-lessons
/plugin install vocabulary-contract
```

## What's inside

```
vocabulary-contract/
├── .claude-plugin/
│   └── plugin.json
├── evals/
│   ├── evals.json       # eval prompts + assertions (skill-creator format)
│   ├── grade.py          # programmatic grader for evals.json's assertions
│   └── trigger-eval.json # description-triggering eval queries
└── skills/
    └── vocabulary-contract/
        ├── SKILL.md
        ├── scripts/check_terms.py
        ├── references/openspec-workflow.md
        ├── assets/agents-md-template.md
        └── test-fixtures/    # known-answer fixtures the evals run against
```

Eval run outputs (per-iteration results, the description-optimization log
and report, packaged `.skill` files) are not committed — they're
regenerated by re-running the skill-creator eval loop against
`evals/evals.json` and are covered by `.gitignore`.
